Design Patterns

设计模式

代码严格遵循标准设计模式，确保高内聚、低耦合、可扩展。共 10 种设计模式（v2.0 新增反滥用责任链模式 + 参数化回调消除竞态）。

Repository

Repository 持久化模式

PersistenceManager

PersistenceManager 作为 Repository 层统一管理 SQLite 数据库，MemoryManager / JournalManager / ConversationHistory / EmotionManager 均作为 Facade 委托给它。
5 张表 + FTS5 全文索引 + WAL 模式，替代 5 个独立 JSON 文件。 所有写操作在 synchronized(lock) 下执行，保证多线程安全。

Facade

门面模式

AiAgentActivity · MemoryManager · JournalManager

三重门面：

AiAgentActivity — 作为 SFW 插件唯一入口，仅负责命令路由（main() 方法中 switch 分发）。 所有业务逻辑委托给 ActivityActions、AiConfig、DeepSeekClient 等核心类处理。

MemoryManager — 记忆操作门面，内部委托 PersistenceManager 的 FTS5 全文搜索 + CRUD。

JournalManager — 日志操作门面，内部委托 PersistenceManager 的 journal 表操作。

调用者无需了解底层 SQLite 实现细节，只需调用 Facade 提供的简单接口。

Singleton

单例模式

AiConfig · StreamPrinter

双组件单例，各自独立：

AiConfig — 全局唯一配置实例，双重检查锁定（DCL）保证线程安全。 管理 API Key、URL、Model、System Prompt 的读写与 Properties 文件持久化。 API Key 使用 AES 加密存储，防止明文泄露。

StreamPrinter — 全局唯一流式输出实例，通过 getInstance() 获取。 内部 daemon 线程在首次实例化时启动，永续运行。所有 Chat/Agent 模式 共享同一 FIFO 队列，无竞态覆盖风险。

Producer-Consumer

生产者-消费者模式

StreamPrinter

API 线程（生产者）将输出通过 offer(char) 或 offerLine() 写入 LinkedBlockingQueue<Item>， 独立 daemon 线程（消费者）从队列取出并以 10ms 间隔逐字输出到控制台。
start() 先全速排空旧队列再启动新流；行输出（系统消息）无延迟直出， 字符输出（AI 流式）10ms 延迟打字机效果。
队列容量无上限，确保高吞吐下不丢字符。

Strategy

策略模式

DeepSeekClient

封装两种 API 调用策略：chatStream()（SSE 流式 + StreamPrinter 逐字输出） 和 chatSync()（标准 JSON 响应 + 完整解析）。 Chat 与 Agent 模式均默认使用流式策略，打字机效果输出。 两种策略共用同一 buildRequestBody() 构建请求体，仅响应解析部分不同。

ReAct

ReAct 模式（推理-行动循环）

AgentExecutor

推理-行动循环：AI 分析任务 → 通过 XML 标签发出指令 → 框架执行并捕获结果 → AI 根据反馈决定下一步。 最多 30 轮循环（execq 上限 50 轮），每轮均为流式输出。支持 12 种 XML 操作标签：
<cmd>、<readfile>、<readdir>、 <sys>、<evaljs>、<eval>、 <web>、<remember>、<download>、 <superise>、<editprompt>、<stop>。
安全模式（ai/exec）下 <cmd> <sys> <readfile> <readdir> <evaljs> <eval> <web> <download> 需 ConfirmationGate 确认；
<remember> <superise> <editprompt> <stop> 为内部/自我表达操作，无需确认。自由模式（ai/execs）自动允许所有。

<stop> 标签（v2.0 新增）：三通道覆盖 — QQ 消息直接检测 + execq AI 输出 + 控制台 TAG_PATTERN， 调用 executeStop() → markStopped() 立即中断 Agent 执行。

Gatekeeper

确认门控模式

ConfirmationGate

统一高危操作确认机制。内部使用 CountDownLatch(1) 实现同步等待。 Agent 线程调用 await(type, desc) 阻塞等待 → 用户通过 ai/yes / ai/no 触发 confirm(boolean) → latch 释放，Agent 继续或跳过。支持 60 秒超时自动拒绝。 setBypassConfirm(true)（execs 模式）跳过所有等待，直接返回 true。 所有状态赋值在 synchronized 块内完成，保证线程安全。

Thread-Safe

EDT 安全路由模式

EdtUtils

SFW 控制台基于 Swing JTextPane，仅 EDT 线程可安全写入。 EdtUtils 作为统一的输出路由层，所有 println() / print() 方法内部路由到单例 StreamPrinter.getInstance()， 通过 offerLine() / offer(char) 将内容送入 FIFO 队列， 由守护线程统一输出到 SFW 控制台。替代此前四处重复的 invokeAndWait 代码， 统一管理、消除竞态。

Companion

情绪状态机 + 陪伴系统模式

EmotionManager · SurpriseWindow

AI 具备类人的情绪系统，影响其行为表现：

情绪检测 — 用户消息与关键词词典匹配，识别辱骂（-20 happiness）、夸赞（+15）、安慰（+25）。
操作反馈 — 每次 Agent 操作成功/失败自动调整 happiness（成功 +3，失败 -8）。
暂停机制 — happiness ≤ -30 或连续 3 次失败触发 Agent 暂停，等待用户安慰或指导。超时 120 秒自动恢复。
彩蛋系统 — happiness ≥ 95 且连续被夸 2 次触发彩蛋弹窗（SurpriseWindow）， 触发后消耗 happiness → 60 防止频繁触发（v2.0 阈值从 80 提升至 95）。 支持撒娇模式（粉色）和惊喜模式（橘色），3-8 秒自动关闭。
提示词自我进化 — AI 可通过 <editprompt> 标签修改自身 system prompt， 实现性格的逐步演化。
持久化 — 情绪状态保存到 app_state("emotion")（SQLite），重启后恢复。暂停状态重启后自动重置。

Chain

反滥用责任链模式

QQMessageHandler

v2.0 新增，好友/群邀请处理中的多层防护机制：

好友申请验证：无验证消息 → 拒绝；有验证消息 → 接受 + 记录时间戳。
5分钟反滥用窗口：好友通过后 5 分钟内发起群邀请 → 立即拒绝 + 发送警告 "请不要厚着脸皮拉我进群"。
通过 ConcurrentHashMap<Long, Long> friendAcceptTimestamps 跟踪时间戳， 异步任务在 5 分钟后自动清理过期记录。
7步群邀请决策链：①主人无条件接受 → ②反滥用检测 → ③好感度查询 → ④背叛者拒绝 → ⑤ >600 接受 → ⑥ 200-600 忽略 → ⑦ ≤200 拒绝。
每个步骤都是独立判断节点，任一命中即终止链，后续步骤不再执行。
并发安全：ConcurrentHashMap + synchronized 块保护决策过程， 防止同一用户的多个邀请并发竞争。
