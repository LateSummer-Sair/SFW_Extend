Sair Music Player V4.1 — 技术总结与功能说明

基于 SairFramework 控制台框架构建的模块化 Java 音乐播放器。 支持 MP3/WAV 解码、15段专业均衡器、环绕声、声场扩展、Viper 综合音效、 LRC 歌词同步、桌面歌词、播放列表持久化等功能。

3. 技术细节

3.1 播放引擎 (SairMP3Player)

全量内存加载方案：

  1. AudioSystem.getAudioInputStream(file) → 解码为 PCM AudioInputStream
  2. putsMemory() → 8KB 分块读取，写入 ByteArrayOutputStream(8MB 初始)，返回 byte[]
  3. 播放时 new ByteArrayInputStream(bais) → 4KB 缓冲循环读取 → processAudio() DSP → line.write()

关键字段：

字段类型说明
baisbyte[]PCM 音频数据（全量）
nowBytePositionint当前播放字节偏移量
nowPositionfloat播放进度 0.0~1.0
maxTimeint总时长（秒）
isContinueboolean暂停标志（false 时 wait）
isBreakboolean停止标志
isSetPosboolean跳转请求标志

时长计算：maxTime = bais.length / (sampleRate × 2 × channels)（PCM signed 16-bit）。

跳转实现：ByteArrayInputStream.reset() 回到开头 → skip(nowBytePosition) 跳到目标位置。由于是内存数据，跳转瞬时完成。

音频处理器（EQ、Volume、Surround 等）由 PlayerActions 统一管理，跨歌曲共享同一个实例。每次切换歌曲时通过 setNewPlayer() 绑定到新播放器，而非创建新处理器。

3.2 音频 DSP 管道

BiquadFilter — 双二阶 IIR 滤波器

支持 7 种滤波类型：PEAKING, LOWPASS, HIGHPASS, LOWSHELF, HIGHSHELF, NOTCH, BANDPASS。采用 RBJ Audio-EQ-Cookbook 公式计算系数。每个样本经过 y[n] = b0·x[n] + b1·x[n-1] + b2·x[n-2] - a1·y[n-1] - a2·y[n-2] 差分方程处理。

SurroundProcessor — 环绕声

  • STEREO_EXPAND：M/S 分解 → 放大 Side 信号 → 重组（核心公式：side *= expandWidth）
  • VIRTUAL_SURROUND：哈斯效应延迟线 + 交叉馈送
  • DIFFERENTIAL：基于左右声道差分的相位处理
  • WIDE_STEREO：高频增强 + 中置图像压缩

SoundstageExpander — 声场扩展

M/S 分解 → 宽度扩展 → 中置图像控制 → 简易混响（100ms 延迟线） → 简化 HRTF 空间化 → 高度增强（高频强调）。参数：width(0.5-3.0)、depth(0.0-2.0)、height(0.5-2.0)、centerImage(0.0-2.0)。

ViperEffectsProcessor — 综合音效

包含 4 个独立子模块：BassEnhancer（谐波增强低频）、TrebleEnhancer（动态高频提升）、Compressor（动态范围压缩，ratio 1.0-4.0）、Exciter（谐波激励器）。支持 6 种预设配置。

3.3 歌词同步

FlushLRCThread 每 200ms 轮询一次：

  1. nowTime = mp3MaxTime × getNowPos()（当前播放秒数）
  2. getApproximate()：从后向前遍历时间数组，找到最后一个 time[i] < nowTime 的索引
  3. flushLRCUI(index, size)：设置 JList 选中项 + 滚动到视图中央

FlushDesktopLRC 复用同一个 LRC 对象（通过 AutoNextPlayThread.creatLRC() 共享），将匹配的歌词行渲染到 DesktopLyricWindow 浮动窗口。

3.4 配置持久化

AudioConfig 以 key=value 格式读写 audio_config.cfg：

  • EQ：eq_band_0 ~ eq_band_14（线性增益值）
  • 音量：volume、master_volume、muted
  • 环绕声：surround_mode、surround_width、surround_mix
  • 声场：soundstage_enabled、soundstage_width、depth、height、center
  • Viper：viper_enabled、viper_bass_enabled/strength、viper_treble_enabled/strength、viper_compressor_enabled/ratio、viper_exciter_enabled/amount
配置 不会自动加载（loadAudioConfig() 已被注释）。用户需通过 EQ 面板或高级音效面板的 Save Config / Load Config 按钮手动操作。

4. 注意事项

4.1 关键约定

音频处理器跨歌曲共享：EQProcessor、VolumeController、SurroundProcessor、SoundstageExpander、ViperEffectsProcessor 由 PlayerActions 统一创建并管理。切换歌曲时不重建，只通过 setNewPlayer() 重新绑定。这保证了 EQ/音效设置在切歌时保持不变。
全量内存加载：SairMP3Player 使用 putsMemory() 将整个 MP3 解码后的 PCM 数据加载进内存（byte[] bais）。好处是跳转瞬时完成、无需处理流生命周期；代价是每首歌曲占用的内存约为 MP3 大小的 4-5 倍（PCM 16-bit 立体声 44.1kHz ≈ 10MB/分钟）。
UI 面板单例缓存：EQPanel 和 AdvancedEffectsPanel 分别只在首次请求时创建一次（null 检查），之后复用。千万不要在 show 方法中去掉 null 检查，否则会产生重复且无响应的面板实例。
ViperEffectsProcessor 的数据复制：process() 方法必须 System.arraycopy 输入数据到新数组再处理，不能直接修改输入引用，否则会导致「同一首歌循环播放时音效错乱」的 Bug。
配置 if-else 链：AudioConfig.loadConfig() 中环绕声和 Viper 的配置项必须用 else if (key.equals(...) && obj != null) 独立判断，不能用嵌套 if-else 结构（否则环绕声 != null 会导致 Viper 分支永远不执行）。

4.2 编码规范

  • 禁止使用外部库：所有功能（MP3 解码、DSP 滤波、LRC 解析）仅依赖 JDK 标准库 + SairFramework
  • FCM 颜色同步：UI 文字颜色必须使用 FCM.EXECTION_pathInfo_Color / FCM.EXECTION_help_Color，禁止硬编码 Color 常量
  • 项目文件编码：UTF-8
  • Java 版本：需支持 try-with-resources、lambda、switch-string（JDK 7+）

4.3 已知陷阱

陷阱原因解决
进度条始终满格 totalBytes 用 MP3 文件大小，currentBytePosition 追 PCM 解码字节（约 5× 差距） 现在用 bais.length（PCM 字节）统一计算
桌面歌词拖拽消失 静态内部类中 setLocation 移动的是 JPanel 而非 JWindow 改为非静态内部类，调用 DesktopLyricWindow.this.setLocation()
Viper 配置加载无效 loadConfig 的 else if 嵌套结构导致 Viper key 永不匹配 改为每个 key 独立判断 + null 检查
重复打开高级面板产生空面板 showAdvancedEffectsPanel 无缓存，每次都 new 添加 advancedEffectsPanel 字段做 null 检查缓存
